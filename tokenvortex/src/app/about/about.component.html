Token Vortex

By Paul Collins

Technical Description
TokenVortex is an Ethereum contract manager and transaction sequence tester. With TokenVortex you can add a contract and run methods similar to Remix. With TokenVortex however, you can set up a sequence of methods to be executed. The sequence can then be run whenever needed and a collection of logs recorded. You can also store a history of contract and test account information using a traditional app user account. TokenVortex makes it easy to test out your Ethereum contract using easy to access accounts and input variables.


Non-Technical Description
A blockchain like Bitcoin is analogous to a bank with one currency. The Ethereum blockchain is analogous to users being able to create their own bank with their own currency and built in virtual bank tellers. Bank teller procedures can be written and sent to the bank. For example:

  1. Provide ID
  2. Open Account
  3. Deposit Funds
These processes have to be extremely secure before being implemented. Invalid or insecure procedures need to be identified for example:

  1. Provide ID
  2. Leave money on counter and go for a walk
  3. Deposit Funds
  4. Open Account
The above is a scenario would fail both from a security and functional perspective.

Without a sequence tester, the procedure can still be tested using tools such as Remix but getting the results from one step and manually using them in the next could end up being a difficult and time consuming task.

Alternatively an 'off-chain' script could be written using a programming language such as javascript just to get these steps to run in order automatically.

Specialists in cryptography and web security however may not have the required programming skills or the time to learn a programming language. TokenVortex automates the testing process without having to implement a programming script. Developers can write and test out the 'bank teller' procedures using 'monopoly money' (test networks) before deploying to the 'bank' (The Ethereum blockchain).


Back Story
Babu Pillai, doctoral candidate at Griffith University Gold Coast working on securing the transfer value between blockchains needed a website to use as an experimental lab. I volunteered to build the web application that would insecurely transfer value but provide a gap between application driven transactions where different cryptographical approaches could be applied.


Technical Specs
  • The client side is an Angular application hosted on Google Cloud via Firebase (free tier).
  • The server side is a Node based Express Generator application hosted on Heroku (free tier).
  • Account and contract data is stored and retrieved from a Mongodb database (free tier).
  • Ethers.js is used to interface with the Ethereum network.
  • The gas for transactions is funded by the admin account.
  • Ether for the test environments can be generated via faucets. ( goerli, rinkeby, ropsten , kovan )
  • The TokenVortex contract was built using inherited ERC20 contracts from OpenZeppelin.
  • The contract was initially compiled using Truffle, flattened using Etherlime, then finally re-compiled and deployed in Remix.
  • The source code is stored in Github. Pull requests welcome.

Backlog Tasks
The app needs more work. Here is a list of the features that will be added. Pull requests welcome.
  • Capture method outputs: Outputs from view method calls need to be captured and stored in a custom field. The outputs are specified in the ABI so the same code used to specify inputs can be used.
  • Push transaction updates: At the moment, a refresh button is supplied to get updates on running transactions. By incorporating RXJS Observables with socket.io, updates will be received as push notifications.
  • Run All: Sequences are currently run manually one method at a time. 'Run all' functionality needs to be added in to fully automate running the sequence.
  • Events: Contract events are also specified in the ABI so capturing these should be similar to capturing 'view' transaction outputs.
  • Results Table:The results are currently displayed as a JSON dump with an Etherscan link. A results table is needed to clean up the display of transaction results.
  • Compiler: Incorporate the solc library to compile contracts
